SCENE 04 · 代理層 Agents

當模型開始自己思考、用工具、分多步完成任務

RAG 是單次「檢索→生成」；代理是它的超集——一個會循環的系統。

★ ReAct 迴圈：思考 → 行動 → 觀察 Agent Loop

整個畫面。代理不是一問一答，而是一個迴圈：模型 思考(Reason)下一步做什麼 → 行動(Act)呼叫一個工具 → 觀察(Observe)看結果 → 再思考……直到完成。步驟由模型自己決定。

harness 撐起這個迴圈；模型是腦，工具是手，觀察結果再餵回。

工具呼叫 / Function Calling。模型怎麼「行動」？它輸出一個結構化請求(例 {"tool":"search_policy","args":{"q":"大額現金門檻"}})，由 harness 執行、把結果回傳。模型自己不跑程式，只發出請求。

MCP(Model Context Protocol)。一個開放標準，讓任何工具/資料源用同一種接口接上任何代理——工具界的「USB 插口」，不必為每個工具寫膠水程式。

skill。打包好的能力：一個有名字、可重用的「提示 + 工具 + 步驟」組合(例：一個『合規查詢 skill』綁好對的提示和政策搜尋工具)。

harness。模型外圍的執行骨架：撐起迴圈、管工具、存記憶/狀態、解析模型的工具請求、執行、把結果餵回。模型是腦，harness 是讓它能行動的身體；沒有 harness，模型只是文字進文字出。
★ The ReAct loop: Reason → Act → Observe Agent Loop

The whole picture. An agent isn't one question, one answer — it's a loop: the model Reasons about what to do next → Acts by calling a tool → Observes the result → reasons again… until it's done. The model decides the steps itself.

The harness holds up the loop; the model is the brain, tools are the hands, results are fed back in.

Tool calls / Function calling. How does the model “act”? It emits a structured request (e.g. {"tool":"search_policy","args":{"q":"large-cash threshold"}}), the harness runs it and feeds the result back. The model never runs code itself — it only issues requests.

MCP (Model Context Protocol). An open standard that lets any tool/data source connect to any agent through one interface — the “USB port” of tools, so you don't write glue code for each one.

skill. A packaged capability: a named, reusable bundle of “prompt + tools + steps” (e.g. a “compliance-lookup skill” wiring the right prompt to a policy-search tool).

harness. The execution scaffold around the model: it runs the loop, manages tools, holds memory/state, parses the model's tool requests, executes them, and feeds results back. The model is the brain; the harness is the body that lets it act — without a harness, a model is just text-in, text-out.

SCENE 05 · 合規視角

四個切面，追問機密資料的去向

在哪、會不會越界、誰能碰、留多久——套在系統圖的每個資料落點上逐一問一遍。

① 靜態 At Rest

落點：向量庫、存的原文塊、(微調過的)模型權重、日誌、備份。

風險：原文以明文存在多處；向量可反推；權重記憶訓練資料且難刪。

控制：靜態加密 + 客戶自管金鑰、權限、可刪除(連衍生物一起清)。

② 傳輸 In Transit

落點：每一跳，尤其 ⑨→⑩「完整 Prompt 送往 MaaS」。

風險：TLS 只擋網絡竊聽，不等於對收方保密；走公有 MaaS = 機密原文出境。

控制：自託管/主權推理；或對送出內容去敏；明確邊界。

③ 使用與日誌 In Use + Logs

落點：推理時的 KV cache；日誌/可觀測平台記下的 prompt+回應。

風險：最易被忽略——機密的持久明文副本；供應商可能拿去訓練或自行保留。

控制：入庫前去敏/遮蔽、保留期上限、合約規範供應商資料用途。

④ 落地與權限 Residency + Access

落點：各儲存與推理的物理司法管轄；RAG 檢索對原文件的權限。

風險：跨境傳輸違規；越權檢索(用戶取到無權看的段落)。

控制：在地落點；chunk 層綁原文件 ACL，檢索時過濾。

兩個要記住的重點：最敏感的一個動作是「機密被拼進 Prompt、再送出去」(⑨→⑩)；最容易被漏的一個落點是日誌。把這四個面當成一張清單，就是「AI 場景的數據安全合規」。
① At Rest

Where: the vector DB, stored source chunks, (fine-tuned) model weights, logs, backups.

Risk: source text sits in plaintext in many places; vectors can be inverted; weights memorize training data and are hard to delete.

Controls: encryption at rest + customer-managed keys, permissions, deletability (purge derivatives too).

② In Transit

Where: every hop, especially ⑨→⑩ “the full Prompt sent to MaaS”.

Risk: TLS only blocks network eavesdropping, not confidentiality from the recipient; a public MaaS = secrets leave your boundary.

Controls: self-hosted / sovereign inference; or de-sensitize what you send; clear boundaries.

③ In Use + Logs

Where: the KV cache during inference; prompts+responses recorded by logs / observability.

Risk: the most overlooked — persistent plaintext copies of secrets; the provider may train on them or retain them.

Controls: de-sensitize / mask before storing, retention caps, contract the provider's use of data.

④ Residency + Access

Where: the physical jurisdiction of each store and of inference; RAG retrieval's permission over source documents.

Risk: cross-border transfer violations; over-privileged retrieval (a user gets passages they shouldn't see).

Controls: in-region placement; bind chunk-level ACLs to the source document and filter at retrieval.

Two things to remember: the most sensitive single action is “secrets get packed into the Prompt and sent out” (⑨→⑩); the most easily-missed single site is the logs. Treat these four lenses as a checklist and you have “data-security compliance for AI.”

整個訣竅：把語意變成幾何。嵌入把一段文字變成高維空間裡的一個「點」(例如 1024 維)。模型被訓練成：意思相近的文字落在相近的位置，意思不同的落得遠。於是「語意」變成「位置」、「語意相似」變成「兩點之間的距離/夾角」。檢索之所以成立就靠這個——你的問題和相關段落意思相近 → 向量相近 → 找最近的鄰居就找到它。

怎麼學會這樣擺？嵌入模型是個神經網絡，用「對比學習(contrastive learning)」訓練：餵它大量「相關的一對」和「不相關的一對」，訓練它把相關的拉近、不相關的推遠。幾百萬次之後，它就學出一套座標系，讓「近 = 相關」。沒有人手工指定那些數字。

每個數字代表什麼？單獨一個維度通常沒有人讀得懂的意義。意思是「分散」在全部維度上的——是整體的位置在編碼語意，不是某一條軸。維度越多，越有空間把大量不同意思分得開、不撞一起。

意思相近的聚成一團；問題落在相關那團旁邊，最近鄰就是最該檢索的段落。

一個比喻：想像一個圖書館，把書擺在地板上，讓主題相近的書互相靠近。要找「跟這本相似的書」，看它旁邊就行。嵌入就是把這件事，在高維空間、對「文字的意思」做一次。

機制回顧：文字 → 分詞 → 過 transformer 編碼器 → 池化 → 正規化。至於「相近」怎麼量、百萬向量怎麼快找——是下一站 T2。

The whole trick: turn meaning into geometry. An embedding turns a piece of text into a single “point” in high-dimensional space (say 1024 dims). The model is trained so that texts with similar meaning land in nearby positions and different meanings land far apart. So “meaning” becomes “position,” and “semantic similarity” becomes “distance/angle between two points.” That's what makes retrieval work — your question and the relevant passage mean similar things → their vectors are close → find the nearest neighbors and you've found it.

How does it learn to arrange things this way? The embedding model is a neural network trained with “contrastive learning”: feed it many “related pairs” and “unrelated pairs,” training it to pull related ones together and push unrelated ones apart. After millions of rounds it grows a coordinate system where “near = related.” No one hand-assigns those numbers.

What does each number mean? A single dimension usually has no human-readable meaning. Meaning is “spread” across all dimensions — it's the overall position that encodes semantics, not any one axis. More dimensions = more room to keep many distinct meanings apart without collisions.

Similar meanings cluster; the question lands beside the relevant cluster, and its nearest neighbors are exactly the chunks to retrieve.

An analogy: imagine a library that lays books on the floor so that books on similar topics sit near each other. To find “books like this one,” just look at its neighbors. An embedding does exactly this — in high-dimensional space, for the “meaning of text.”

Mechanism recap: text → tokenize → pass through a transformer encoder → pool → normalize. How “closeness” is measured and how millions of vectors are searched fast is the next stop, T2.

 {{ tk.deepEmbed.text }}

承上。T1 講了「意思相近 = 位置相近」，但「相近」要有精確定義，而且要在百萬條向量裡快速找到最近的幾條。這站答兩件事：怎麼量、怎麼快。

① 相似度怎麼量。意思編碼在向量「指向哪個方向」。主流量度是「餘弦相似度」——兩向量的夾角：同方向=1、垂直=0、反方向=−1。嵌入向量通常已正規化(長度為 1)，這時內積就等於餘弦——系統就是把問題向量和每條文件向量做內積，得出分數(就是那個 score 0.89)。

問題  q  = [ 0.6,  0.8,  0.0]
相關 d1  = [ 0.5,  0.85, 0.1]   q·d1 = .30+.68+.00 = 0.98  ← 高，相似
無關 d2  = [-0.7, 0.1,  0.7]   q·d2 = -.42+.08+.00 = -0.34  ← 低(反向)

相似度 = 兩個箭頭有多「同方向」；正規化後，算內積就等於算餘弦。

② 百萬條裡怎麼快找。笨方法是問題和「每一條」都算一次再排序——1000 萬 × 1024 維上百億次乘加，即時回答撐不住。解法是「ANN 近似最近鄰」：用聰明的索引只跟一小部分比較，用一點點準確度換幾百倍速度。

為何要「近似」？高維空間裡保證「精確」最近鄰幾乎等於要檢查全部。所以接受 95–99% 召回，換巨大加速。對 RAG，速度才是命。

ANN：先把向量粗分群，只搜離問題最近的那幾群，其餘整群跳過。

兩個主流做法：HNSW——把向量連成「鄰居圖」，像「朋友的朋友」幾跳就到目標附近(最常用)；IVF——用 k-means 切群，只搜最近的幾群。FAISS 和各向量庫底層就是這些。

下一站 T3：段落塞進 prompt 後，模型推理(prefill / decode / KV cache)怎麼跑、為何省算力。

Continuing. T1 said “similar meaning = nearby position,” but “nearby” needs a precise definition — and you must find the nearest few among millions of vectors fast. This stop answers two things: how to measure, and how to do it quickly.

① Measuring similarity. Meaning is encoded in which direction a vector points. The standard measure is “cosine similarity” — the angle between two vectors: same direction = 1, perpendicular = 0, opposite = −1. Embedding vectors are usually normalized (length 1), so the dot product equals the cosine — the system takes the dot product of the query vector with every document vector to get a score (that 0.89).

query     q  = [ 0.6,  0.8,  0.0]
related   d1 = [ 0.5,  0.85, 0.1]   q·d1 = .30+.68+.00 = 0.98  ← high, similar
unrelated d2 = [-0.7, 0.1,  0.7]   q·d2 = -.42+.08+.00 = -0.34  ← low (opposite)

Similarity = how much two arrows point the same way; once normalized, the dot product equals the cosine.

② Finding the nearest fast among millions. The naive way scores the query against “every” vector then sorts — 10M × 1024 dims is tens of billions of multiply-adds, too slow for real time. The fix is “ANN — approximate nearest neighbor”: a smart index compares against only a small subset, trading a little accuracy for hundreds of times the speed.

Why “approximate”? In high-dim space, guaranteeing the “exact” nearest neighbor is almost the same as checking all of them. So accept 95–99% recall for a huge speedup. For RAG, speed is everything.

ANN: coarsely cluster the vectors first, search only the few clusters nearest the query, skip whole clusters.

Two mainstream methods: HNSW — link vectors into a “neighbor graph” and hop “friend of a friend” to the target's vicinity in a few jumps (most common); IVF — k-means into clusters, search only the nearest few. FAISS and the vector DBs are built on these.

Next stop, T3: once chunks are stuffed into the prompt, how the model's inference (prefill / decode / KV cache) runs and why it saves compute.

 {{ tk.deepRetrieval.text }}

整個畫面。Prompt 組好後，模型逐字生成答案(自迴歸)，分兩段，成本差很大。

Prefill(預填充)。一次並行讀入整個 prompt，逐層算出每個 token 的 key/value，存進 KV cache。計算密集但並行 → 每 token 快。

Decode(解碼)。逐個生成輸出 token，每個新字都要看前面所有 token。序列式(不能並行)，受記憶體頻寬限制 → 慢；答案越長越久。

prefill 把整段 prompt 一次算好；decode 逐字生成，每字只算新格、重用 cache。

KV cache 為何省算力。沒快取的話，生成第 n 字要把前面 n−1 個的 K/V 重算一次，巨量重複(O(n²))。KV cache 把已算的 K/V 存起來，生成新字時只算新這一個、其餘重用。代價：cache 很吃 GPU 記憶體，才有 PagedAttention(vLLM)這類優化。

收束。長 prompt 更貴(prefill 大、cache 大)；長答案更慢(decode 序列式)。這就是推理成本與延遲的來源。

The whole picture. Once the Prompt is assembled, the model generates the answer token by token (autoregressive), in two phases whose costs differ greatly.

Prefill. Read the whole prompt at once in parallel, computing each token's key/value layer by layer and storing them in the KV cache. Compute-heavy but parallel → fast per token.

Decode. Generate output tokens one at a time; each new token must attend to all previous ones. Sequential (can't parallelize), memory-bandwidth bound → slow; the longer the answer, the longer it takes.

Prefill computes the whole prompt at once; decode generates token by token, computing only the new one and reusing the cache.

Why the KV cache saves compute. Without a cache, generating the n-th token recomputes the K/V of all n−1 previous ones — massive duplication (O(n²)). The KV cache stores already-computed K/V, so a new token computes only itself and reuses the rest. The cost: the cache eats GPU memory, which is why optimizations like PagedAttention (vLLM) exist.

Wrap-up. A long prompt is more expensive (big prefill, big cache); a long answer is slower (sequential decode). That's where inference cost and latency come from.

是什麼。模型在給最終答案前，先生成一段「思考」(chain-of-thought)，用推理時的額外算力換準確度。代表：OpenAI o 系列、各家的 extended thinking。

為何關鍵。2026 最大的轉變——從「一次過吐答案」變成「先想再答」。難題準確度大升；代價是更慢、更貴。

接你學過的。「思考」也是 token，decode 階段吐的 token 變多 → 成本與延遲都更高。這也是為何「推理優化/快取」在 2026 更重要。

先生成一段「思考」再給答案——用推理時的算力換準確度。

What it is. Before giving its final answer, the model first generates a stretch of “thinking” (chain-of-thought), trading extra inference-time compute for accuracy. Examples: OpenAI's o-series, various vendors' extended thinking.

Why it matters. 2026's biggest shift — from “blurt the answer in one pass” to “think first, then answer.” Accuracy on hard problems jumps; the cost is slower and pricier.

Connecting to what you learned. “Thinking” is also tokens; more tokens emitted during decode → higher cost and latency. That's exactly why “inference optimization / caching” matters more in 2026.

Generate a stretch of “thinking” first, then the answer — trading inference compute for accuracy.

是什麼。經典 RAG 是「一次檢索 → 生成」(固定管線)。代理式檢索(Agentic RAG)讓模型自己決定何時搜、讀完不夠再搜、多輪逼近答案。脈絡工程(Context Engineering)是 prompt engineering 的升級：重點是管理整個上下文窗裡放什麼——檢索結果、記憶、工具輸出、對話歷史。

為何關鍵。這才是 RAG 在 2026 的實際形態：檢索是 agent 會呼叫的一個工具，不是死管線。地圖裡「RAG 管線」與「代理層」的分界，在前沿其實是融合的。

What it is. Classic RAG is “retrieve once → generate” (a fixed pipeline). Agentic RAG lets the model decide when to search, search again if what it read isn't enough, and converge on the answer over multiple rounds. Context engineering is prompt engineering leveled up: the focus is managing what goes into the whole context window — retrieved results, memory, tool outputs, conversation history.

Why it matters. This is what RAG actually looks like in 2026: retrieval is a tool the agent calls, not a dead pipeline. On this map, the boundary between “the RAG pipeline” and “the agent layer” is, at the frontier, blurred.

 {{ tk.fArag.text }}

是什麼。SFT 教模型「照示範做」；偏好微調再教它「人類較喜歡哪一個答案」。RLHF：用人類對答案的排序訓一個「獎勵模型」，再用它優化主模型。DPO：跳過獎勵模型，直接從「偏好對」學，更簡單穩定。

為何關鍵。這是你「改行為」管線缺的另一半——現代模型的「聽話、有禮、安全」大多來自這一層，而非 SFT。

What it is. SFT teaches the model to “follow demonstrations”; preference tuning then teaches it “which answer humans prefer.” RLHF: train a “reward model” from human rankings of answers, then use it to optimize the main model. DPO: skip the reward model and learn directly from “preference pairs” — simpler and more stable.

Why it matters. This is the other half missing from your “change behavior” pipeline — modern models' “obedient, polite, safe” behavior mostly comes from this layer, not from SFT.

 {{ tk.fPref.text }}

是什麼。把模型分成很多「專家」子網路，每個 token 只激活其中幾個，而不是跑整個模型；一個「路由器」決定每個 token 交給哪些專家。

為何關鍵。讓總參數量可以極大(萬億級)，但每次運算只用一小部分 → 又大又相對便宜。2026 多數前沿大模型的底層架構。

路由器只挑幾個專家來算，其餘不動——所以又大又便宜。

What it is. Split the model into many “expert” sub-networks; each token activates only a few of them instead of running the whole model; a “router” decides which experts each token goes to.

Why it matters. It lets total parameters grow enormous (trillions) while each computation uses only a small fraction → huge yet relatively cheap. The underlying architecture of most frontier models in 2026.

The router picks just a few experts to run, leaving the rest idle — so it's both big and cheap.

 {{ tk.fMoe.text }}

提示快取(Prompt / Prefix Caching)。跨請求重用「相同前綴」的 KV cache——例如固定的 system prompt + 一份長文件，第二次就不必重算它的 prefill，直接大砍成本與延遲。

推測解碼(Speculative Decoding)。用一個小模型快速「猜」未來幾個 token，大模型一次過驗證；猜中就省掉逐字的時間，猜錯就退回。加速生成而不改變輸出。

為何關鍵。推理模型讓 token 變多、變貴，這兩招是 2026 把推理拉回「便宜又快」的主力。

用小模型先猜一批，大模型一次過驗證——猜中就省掉逐字的時間。

Prompt / Prefix caching. Reuse the KV cache of a “shared prefix” across requests — e.g. a fixed system prompt + a long document; the second time you skip recomputing its prefill, slashing cost and latency.

Speculative decoding. A small model quickly “guesses” the next few tokens; the big model verifies them in one pass; correct guesses skip token-by-token time, wrong ones roll back. Speeds up generation without changing the output.

Why it matters. Reasoning models make tokens more numerous and pricier; these two are the main levers in 2026 for pulling inference back to “cheap and fast.”

Let a small model guess a batch first, the big model verifies in one pass — correct guesses save token-by-token time.

 {{ tk.fInfra.text }}

多代理(Multi-agent)。單一 ReAct 迴圈之外，把任務拆給多個專責 agent：一個「主管」分派，多個「worker」各做一塊，彼此交接結果。

記憶(Memory)。短期=上下文窗內的近期內容；長期=外部記憶庫，用向量檢索把相關往事取回(把 RAG 用在「自己的歷史」上)。

為何關鍵。複雜、長流程的任務單一 agent 撐不住；2026 的 agent 系統多是多代理 + 記憶。

主管分派 → 各 worker 做一塊 → 交接整合。

Multi-agent. Beyond a single ReAct loop, split the task across several specialized agents: a “supervisor” assigns work, multiple “workers” each handle a piece, and they hand off results to one another.

Memory. Short-term = recent content within the context window; long-term = an external memory store, using vector retrieval to pull back relevant past events (RAG applied to “its own history”).

Why it matters. A single agent can't carry complex, long-running tasks; 2026's agent systems are mostly multi-agent + memory.

Supervisor assigns → each worker does a piece → hand off and integrate.

 {{ tk.fMulti.text }}

是什麼。攻擊者把惡意指令藏在「會被模型讀到的內容」裡，劫持模型去做不該做的事。直接注入=藏在用戶輸入；間接注入=藏在被檢索的文件、網頁、或工具回傳裡——模型分不清「資料」與「指令」。

為何關鍵。這是 RAG/agent 獨有、2026 的頭等安全風險。尤其「間接注入」：任何會檢索外部內容的系統都中招——被你檢索進來的那段文字，本身就可能是攻擊。

惡意指令可藏在用戶輸入(直接)，也可藏在被檢索的內容裡(間接)。

What it is. An attacker hides malicious instructions inside “content the model will read,” hijacking it into doing what it shouldn't. Direct injection = hidden in user input; indirect injection = hidden in retrieved documents, web pages, or tool outputs — the model can't tell “data” from “instructions.”

Why it matters. This is unique to RAG/agents — a top security risk in 2026. Especially indirect injection: any system that retrieves external content is exposed — the very text you retrieve may itself be the attack.

Malicious instructions can hide in user input (direct) or in retrieved content (indirect).

 {{ tk.fSec.text }}

是什麼。你怎麼知道系統在工作？RAGAS：一套指標量 RAG 的「忠實度、檢索相關性、答案相關性」。LLM-as-Judge：用另一個強模型按準則替答案打分，取代昂貴緩慢的人工評估。

What it is. How do you know the system works? RAGAS: a set of metrics for RAG's “faithfulness, retrieval relevance, answer relevance.” LLM-as-Judge: use another strong model to score answers against criteria, replacing slow, expensive human evaluation.

Why it matters. Evaluation is the crux of applied work — “how you know it's right and where it breaks” is worth more than any single technique. In practice, most RAG failures come from bad retrieval, and evaluation is what catches that.
