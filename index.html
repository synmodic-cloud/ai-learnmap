SCENE 04 · 代理層 Agents

當模型開始自己思考、用工具、分多步完成任務

RAG 是單次「檢索→生成」；代理是它的超集——一個會循環的系統。

★ ReAct 迴圈：思考 → 行動 → 觀察 Agent Loop

整個畫面。代理不是一問一答，而是一個迴圈：模型 思考(Reason)下一步做什麼 → 行動(Act)呼叫一個工具 → 觀察(Observe)看結果 → 再思考……直到完成。步驟由模型自己決定。

harness 撐起這個迴圈；模型是腦，工具是手，觀察結果再餵回。

工具呼叫 / Function Calling。模型怎麼「行動」？它輸出一個結構化請求(例 {"tool":"search_policy","args":{"q":"大額現金門檻"}})，由 harness 執行、把結果回傳。模型自己不跑程式，只發出請求。

MCP(Model Context Protocol)。一個開放標準，讓任何工具/資料源用同一種接口接上任何代理——工具界的「USB 插口」，不必為每個工具寫膠水程式。

skill。打包好的能力：一個有名字、可重用的「提示 + 工具 + 步驟」組合(例：一個『合規查詢 skill』綁好對的提示和政策搜尋工具)。

harness。模型外圍的執行骨架：撐起迴圈、管工具、存記憶/狀態、解析模型的工具請求、執行、把結果餵回。模型是腦，harness 是讓它能行動的身體；沒有 harness，模型只是文字進文字出。

SCENE 05 · 合規視角

四個切面，追問機密資料的去向

在哪、會不會越界、誰能碰、留多久——套在系統圖的每個資料落點上逐一問一遍。

① 靜態 At Rest

落點：向量庫、存的原文塊、(微調過的)模型權重、日誌、備份。

風險：原文以明文存在多處；向量可反推；權重記憶訓練資料且難刪。

控制：靜態加密 + 客戶自管金鑰、權限、可刪除(連衍生物一起清)。

② 傳輸 In Transit

落點：每一跳，尤其 ⑨→⑩「完整 Prompt 送往 MaaS」。

風險：TLS 只擋網絡竊聽，不等於對收方保密；走公有 MaaS = 機密原文出境。

控制：自託管/主權推理；或對送出內容去敏；明確邊界。

③ 使用與日誌 In Use + Logs

落點：推理時的 KV cache；日誌/可觀測平台記下的 prompt+回應。

風險：最易被忽略——機密的持久明文副本；供應商可能拿去訓練或自行保留。

控制：入庫前去敏/遮蔽、保留期上限、合約規範供應商資料用途。

④ 落地與權限 Residency + Access

落點：各儲存與推理的物理司法管轄；RAG 檢索對原文件的權限。

風險：跨境傳輸違規；越權檢索(用戶取到無權看的段落)。

控制：在地落點；chunk 層綁原文件 ACL，檢索時過濾。

兩個要記住的重點：最敏感的一個動作是「機密被拼進 Prompt、再送出去」(⑨→⑩)；最容易被漏的一個落點是日誌。把這四個面當成一張清單，就是「AI 場景的數據安全合規」。

整個訣竅：把語意變成幾何。嵌入把一段文字變成高維空間裡的一個「點」(例如 1024 維)。模型被訓練成：意思相近的文字落在相近的位置，意思不同的落得遠。於是「語意」變成「位置」、「語意相似」變成「兩點之間的距離/夾角」。檢索之所以成立就靠這個——你的問題和相關段落意思相近 → 向量相近 → 找最近的鄰居就找到它。

怎麼學會這樣擺？嵌入模型是個神經網絡，用「對比學習(contrastive learning)」訓練：餵它大量「相關的一對」和「不相關的一對」，訓練它把相關的拉近、不相關的推遠。幾百萬次之後，它就學出一套座標系，讓「近 = 相關」。沒有人手工指定那些數字。

每個數字代表什麼？單獨一個維度通常沒有人讀得懂的意義。意思是「分散」在全部維度上的——是整體的位置在編碼語意，不是某一條軸。維度越多，越有空間把大量不同意思分得開、不撞一起。

意思相近的聚成一團；問題落在相關那團旁邊，最近鄰就是最該檢索的段落。

一個比喻：想像一個圖書館，把書擺在地板上，讓主題相近的書互相靠近。要找「跟這本相似的書」，看它旁邊就行。嵌入就是把這件事，在高維空間、對「文字的意思」做一次。

機制回顧：文字 → 分詞 → 過 transformer 編碼器 → 池化 → 正規化。至於「相近」怎麼量、百萬向量怎麼快找——是下一站 T2。

{{ tk.deepEmbed.text }}

承上。T1 講了「意思相近 = 位置相近」，但「相近」要有精確定義，而且要在百萬條向量裡快速找到最近的幾條。這站答兩件事：怎麼量、怎麼快。

① 相似度怎麼量。意思編碼在向量「指向哪個方向」。主流量度是「餘弦相似度」——兩向量的夾角：同方向=1、垂直=0、反方向=−1。嵌入向量通常已正規化(長度為 1)，這時內積就等於餘弦——系統就是把問題向量和每條文件向量做內積，得出分數(就是那個 score 0.89)。

問題  q  = [ 0.6,  0.8,  0.0]
相關 d1  = [ 0.5,  0.85, 0.1]   q·d1 = .30+.68+.00 = 0.98  ← 高，相似
無關 d2  = [-0.7, 0.1,  0.7]   q·d2 = -.42+.08+.00 = -0.34  ← 低(反向)

相似度 = 兩個箭頭有多「同方向」；正規化後，算內積就等於算餘弦。

② 百萬條裡怎麼快找。笨方法是問題和「每一條」都算一次再排序——1000 萬 × 1024 維上百億次乘加，即時回答撐不住。解法是「ANN 近似最近鄰」：用聰明的索引只跟一小部分比較，用一點點準確度換幾百倍速度。

為何要「近似」？高維空間裡保證「精確」最近鄰幾乎等於要檢查全部。所以接受 95–99% 召回，換巨大加速。對 RAG，速度才是命。

ANN：先把向量粗分群，只搜離問題最近的那幾群，其餘整群跳過。

兩個主流做法：HNSW——把向量連成「鄰居圖」，像「朋友的朋友」幾跳就到目標附近(最常用)；IVF——用 k-means 切群，只搜最近的幾群。FAISS 和各向量庫底層就是這些。

下一站 T3：段落塞進 prompt 後，模型推理(prefill / decode / KV cache)怎麼跑、為何省算力。

{{ tk.deepRetrieval.text }}

整個畫面。Prompt 組好後，模型逐字生成答案(自迴歸)，分兩段，成本差很大。

Prefill(預填充)。一次並行讀入整個 prompt，逐層算出每個 token 的 key/value，存進 KV cache。計算密集但並行 → 每 token 快。

Decode(解碼)。逐個生成輸出 token，每個新字都要看前面所有 token。序列式(不能並行)，受記憶體頻寬限制 → 慢；答案越長越久。

prefill 把整段 prompt 一次算好；decode 逐字生成，每字只算新格、重用 cache。

KV cache 為何省算力。沒快取的話，生成第 n 字要把前面 n−1 個的 K/V 重算一次，巨量重複(O(n²))。KV cache 把已算的 K/V 存起來，生成新字時只算新這一個、其餘重用。代價：cache 很吃 GPU 記憶體，才有 PagedAttention(vLLM)這類優化。

收束。長 prompt 更貴(prefill 大、cache 大)；長答案更慢(decode 序列式)。這就是推理成本與延遲的來源。

{{ tk.deepInfer.text }}

是什麼。模型在給最終答案前，先生成一段「思考」(chain-of-thought)，用推理時的額外算力換準確度。代表：OpenAI o 系列、各家的 extended thinking。

為何關鍵。2026 最大的轉變——從「一次過吐答案」變成「先想再答」。難題準確度大升；代價是更慢、更貴。

接你學過的。「思考」也是 token，decode 階段吐的 token 變多 → 成本與延遲都更高。這也是為何「推理優化/快取」在 2026 更重要。

先生成一段「思考」再給答案——用推理時的算力換準確度。

{{ tk.fReason.text }}

是什麼。經典 RAG 是「一次檢索 → 生成」(固定管線)。代理式檢索(Agentic RAG)讓模型自己決定何時搜、讀完不夠再搜、多輪逼近答案。脈絡工程(Context Engineering)是 prompt engineering 的升級：重點是管理整個上下文窗裡放什麼——檢索結果、記憶、工具輸出、對話歷史。

為何關鍵。這才是 RAG 在 2026 的實際形態：檢索是 agent 會呼叫的一個工具，不是死管線。地圖裡「RAG 管線」與「代理層」的分界，在前沿其實是融合的。

{{ tk.fArag.text }}

是什麼。SFT 教模型「照示範做」；偏好微調再教它「人類較喜歡哪一個答案」。RLHF：用人類對答案的排序訓一個「獎勵模型」，再用它優化主模型。DPO：跳過獎勵模型，直接從「偏好對」學，更簡單穩定。

為何關鍵。這是你「改行為」管線缺的另一半——現代模型的「聽話、有禮、安全」大多來自這一層，而非 SFT。

{{ tk.fPref.text }}

是什麼。把模型分成很多「專家」子網路，每個 token 只激活其中幾個，而不是跑整個模型；一個「路由器」決定每個 token 交給哪些專家。

為何關鍵。讓總參數量可以極大(萬億級)，但每次運算只用一小部分 → 又大又相對便宜。2026 多數前沿大模型的底層架構。

路由器只挑幾個專家來算，其餘不動——所以又大又便宜。

{{ tk.fMoe.text }}

提示快取(Prompt / Prefix Caching)。跨請求重用「相同前綴」的 KV cache——例如固定的 system prompt + 一份長文件，第二次就不必重算它的 prefill，直接大砍成本與延遲。

推測解碼(Speculative Decoding)。用一個小模型快速「猜」未來幾個 token，大模型一次過驗證；猜中就省掉逐字的時間，猜錯就退回。加速生成而不改變輸出。

為何關鍵。推理模型讓 token 變多、變貴，這兩招是 2026 把推理拉回「便宜又快」的主力。

用小模型先猜一批，大模型一次過驗證——猜中就省掉逐字的時間。

{{ tk.fInfra.text }}

多代理(Multi-agent)。單一 ReAct 迴圈之外，把任務拆給多個專責 agent：一個「主管」分派，多個「worker」各做一塊，彼此交接結果。

記憶(Memory)。短期=上下文窗內的近期內容；長期=外部記憶庫，用向量檢索把相關往事取回(把 RAG 用在「自己的歷史」上)。

為何關鍵。複雜、長流程的任務單一 agent 撐不住；2026 的 agent 系統多是多代理 + 記憶。

主管分派 → 各 worker 做一塊 → 交接整合。

{{ tk.fMulti.text }}

是什麼。攻擊者把惡意指令藏在「會被模型讀到的內容」裡，劫持模型去做不該做的事。直接注入=藏在用戶輸入；間接注入=藏在被檢索的文件、網頁、或工具回傳裡——模型分不清「資料」與「指令」。

為何關鍵。這是 RAG/agent 獨有、2026 的頭等安全風險。尤其「間接注入」：任何會檢索外部內容的系統都中招——被你檢索進來的那段文字，本身就可能是攻擊。

惡意指令可藏在用戶輸入(直接)，也可藏在被檢索的內容裡(間接)。

{{ tk.fSec.text }}

是什麼。你怎麼知道系統在工作？RAGAS：一套指標量 RAG 的「忠實度、檢索相關性、答案相關性」。LLM-as-Judge：用另一個強模型按準則替答案打分，取代昂貴緩慢的人工評估。

為何關鍵。評估是 applied 的命門——「怎麼知道它對、它在哪裡崩」比任何單一技術都值錢。實務上 RAG 出事多在檢索錯，評估就是抓這個。

{{ tk.fEval.text }}
